P4 HWPX 125목표 보정(goalfix)

데이터가 어떻게 바뀌었는지 한 번에 보기

검색(retrieval)과 생성(generation)이 같은 데이터를 다르게 쓰기 때문에, 이번 업데이트는 “많이 저장”보다 “의미를 정확히 구분”하는 쪽에 집중했습니다.

22,994최종 청크(chunk) 수
610새로 보강한 청크(chunk)
64.11 MiB검색용 제이슨엘(JSONL) 크기

1. 왜 다시 보강했나

생성(generation) 결과를 검토해보니, 정답 문서는 찾아왔는데 금액의 의미를 잘못 해석하거나, 요구사항처럼 긴 문맥이 필요한 질문에서 답변이 흐려지는 문제가 있었습니다.

핵심은 “숫자가 있다”가 아니라 “이 숫자가 실제 사업예산인지, 자격요건 기준금액인지, 지급조건인지”를 구분하는 것입니다.

2. 데이터 흐름

원본 문서한글(HWP)을 한글 XML(HWPX)로 변환해 문단과 표를 더 안정적으로 분리합니다.
구조화일반 텍스트(text), 표(table), 핵심 후보(fact_candidates)를 나눕니다.
의미 보강금액, 기간, 요구사항, 사업범위 같은 정보를 별도 청크(chunk)로 한 번 더 만듭니다.
크로마(Chroma)content를 문서(documents), metadata를 메타데이터(metadatas)로 적재합니다.
답변 생성질문 유형에 맞는 근거를 골라 대형 언어 모델(LLM)에 넘깁니다.

4. 주요 key 설명

json key의미
chunk_id검색 단위의 고유 식별자입니다. 크로마(Chroma)의 ids로 들어갑니다.
content실제로 임베딩(embedding)되는 검색용 텍스트입니다.
metadata필터와 해석에 필요한 짧은 정보입니다. 긴 원문은 넣지 않습니다.
chunk_typetext, table, fact_candidates, toc 중 하나입니다.
fact_typeproject_budget, threshold_budget, requirements처럼 정보의 의미를 구분합니다.
amount_krw금액을 원 단위 숫자로 정규화한 값입니다.
amount_type금액의 의미입니다. project_budget과 threshold_budget을 구분합니다.
budget_answer_enabled예산 질문의 최종 답변에 써도 되는 금액인지 표시합니다.
answer_policy이 청크를 어떤 질문 유형에서 답변 근거로 쓸 수 있는지 적은 정책입니다.
source_ref검색 결과에서 원문 근거를 다시 찾기 위한 연결 정보입니다.

5. 이번 업데이트에서 특히 중요한 점

구분이전 위험이번 보강
금액모든 금액이 사업예산처럼 보일 수 있음amount_type과 budget_answer_enabled로 답변 가능 여부를 분리
문서 요약document_summary 안의 숫자가 최종 답변으로 쓰일 위험요약은 라우팅용으로 제한하고, 최종 예산은 typed fact를 우선
고차원 질문사업범위, 배경, 요구사항 질문에서 근거가 흩어짐project_scope, project_background, requirements 청크 추가
표표 안의 핵심 값이 긴 원문 속에 묻혀 검색 결과에서 약하게 잡힐 수 있음표 안의 금액, 요구사항, 사업범위 같은 핵심 정보를 fact_candidates로 한 번 더 끌어올려 검색과 생성에서 쓰기 쉽게 보강
정책 keyanswer_policy, budget_answer_enabled는 데이터에만 있으면 자동 적용되지 않음검색 결과를 LLM에 넘기기 전에 context builder 또는 prompt에서 이 값을 읽어 “최종 예산 답변에 쓸 수 있는 근거만 우선 사용”하도록 적용해야 함

6. metadata와 answer_policy는 어떻게 써야 하나

왜 금액 타입을 나눴나

예를 들어 아래 세 문장은 모두 금액을 포함합니다.

1. 사업예산: 1,515,000천원
2. 최근 3년 이내 단일 실적 2억원 이상 보유
3. 선금은 계약금액의 70% 이내 지급 가능

사람이 보면 1번은 실제 사업예산, 2번은 입찰참가자격 기준금액, 3번은 지급조건이라는 걸 구분할 수 있습니다. 하지만 검색된 문장만 LLM에 그대로 넣으면, 2억원이나 70% 같은 숫자를 실제 사업예산처럼 잘못 답할 수 있습니다.

금액 타입을 이렇게 세분화했습니다

금액 타입의미사업예산 답변에 사용
project_budget실제 사업예산으로 답변에 써도 되는 금액사용 가능
total_allocation전체 배정액 또는 총액 성격의 금액질문 의도에 따라 사용 가능
threshold_budget입찰참가자격, 실적 기준 등에 쓰이는 기준금액사업예산으로 사용 금지
payment_terms선금, 중도금, 잔금 같은 지급조건 금액사업예산으로 사용 금지
estimated_price추정가격보조 근거로만 사용
base_amount기초금액보조 근거로만 사용
project_budget

실제 사업예산입니다. 예산 질문, 예산 비교, 차액 계산에 최종 답변 근거로 사용할 수 있습니다.

threshold_budget

입찰참가자격이나 실적 기준에 쓰이는 금액입니다. 중요한 정보지만 사업예산 답변에는 쓰면 안 됩니다.

payment_terms

선금, 중도금, 잔금 같은 지급조건입니다. 계약 조건 설명에는 필요하지만, 사업예산 답변에는 쓰면 안 됩니다.

metadata는 크로마(Chroma)가 검색 결과와 함께 돌려주는 짧은 설명값입니다. 다만 이 값은 대형 언어 모델(LLM)이 자동으로 읽고 판단하는 값이 아닙니다.

핵심: Chroma가 documents와 metadatas를 돌려주면, 우리 코드가 먼저 metadata를 읽고 질문 유형에 맞는 근거를 고른 뒤, 그 선택된 근거와 규칙을 prompt에 넣어야 합니다.

context builder에서 적용하는 방식

def pick_context(question_type, retrieved_rows):
    selected = []
    for row in retrieved_rows:
        meta = row["metadata"]

        if question_type == "budget":
            if meta.get("budget_answer_enabled") is True:
                selected.append(row)
            elif meta.get("fact_type") in ["threshold_budget", "payment_terms"]:
                continue

        elif question_type == "eligibility":
            if meta.get("fact_type") in ["eligibility", "threshold_budget"]:
                selected.append(row)

        elif question_type == "requirements":
            if meta.get("fact_type") in ["requirements", "project_scope"] or meta.get("chunk_type") == "table":
                selected.append(row)

    return selected

7. prompt에는 이렇게 넣는 것이 좋다

LLM에게 JSONL 전체를 그대로 던지는 것이 아니라, 코드가 고른 근거와 해석 규칙을 같이 넘기는 방식이 안전합니다.

[질문]
GKL 그룹웨어 시스템 구축 용역의 사업예산은 얼마입니까?

[선택된 근거]
- evidence_id: EV_xxxx
- source_file: 그랜드코리아레저(주)_2024년도 GKL 그룹웨어 시스템 구축 용역.hwp
- fact_type: project_budget
- amount_krw: 1515000000
- answer_policy: allow_as_project_budget
- content: 사업예산 : 1,515,000천원 | KRW: 1515000000 | budget_type: project_budget

[답변 규칙]
- 예산 질문에서는 budget_answer_enabled=true인 근거를 우선 사용한다.
- threshold_budget은 입찰 자격 기준금액이므로 사업예산 최종값으로 쓰지 않는다.
- payment_terms는 지급조건이므로 사업예산 최종값으로 쓰지 않는다.
- amount_krw가 있으면 금액 계산에는 이 정규화 값을 우선 사용한다.
- 근거가 부족하면 확인되지 않는다고 답한다.
정리: metadata에 정규화된 금액과 세분화된 금액 타입을 넣어둔 것은 맞습니다. 하지만 LLM이 자동으로 이해하는 것은 아니므로, context builder와 prompt에서 이 key를 명시적으로 사용해야 합니다.
