RFP RAG Corpus Design Brief

검색은 넓게, 답변은 엄격하게: RFP 문서용 JSON 구조화 설계

이 corpus는 단순히 문서를 1,000자씩 자른 파일이 아니라, RFP 도메인의 질문 패턴을 고려해 문서 식별 신호, 핵심 후보값, 표 구조, 답변 사용 정책을 JSON metadata로 분리한 retrieval/generation 실험용 데이터입니다.

1. 설계 의도

RFP RAG에서 가장 큰 실패는 정답 문서를 못 찾는 것과, 찾은 문서 안의 잘못된 숫자를 답으로 쓰는 것입니다.

Retrieval

정답 문서를 놓치지 않기

기관명, 사업명, 파일명, 공고번호, 사업유형, alias를 별도 신호로 만들어 문서 회수율을 올립니다.

Grounding

답변 근거를 분리하기

예산, 기간, 제출서류, 자격요건 같은 후보값을 fact chunk로 분리해 LLM이 긴 문서 중간을 덜 헤매게 합니다.

Guardrail

잘못된 값 사용 막기

실적 기준 금액, 지급조건, 공란 서식의 숫자가 사업예산으로 둔갑하지 않도록 answer policy를 metadata에 둡니다.

2. RFP 문서 특징과 주목 키워드

RFP는 사업 설명서이면서 동시에 입찰 안내서, 제출서류 목록, 평가표, 서식 모음입니다. 그래서 같은 값도 문맥에 따라 의미가 달라집니다.

핵심 정보가 흩어져 있음 사업예산, 기간, 제출서류, 평가기준이 한 섹션에만 있지 않고 본문·표·별지·서식에 나뉘어 등장합니다.
표 안의 텍스트가 중요함 공공문서 특성상 긴 문장을 표 박스에 넣는 경우가 많아, 표를 버리면 과업범위와 제출서류를 놓칠 수 있습니다.
같은 숫자의 의미가 다름 10억은 사업예산일 수도, 입찰참가 실적 기준일 수도, 가격제안서 작성 예시일 수도 있습니다.
기간도 여러 종류가 있음 사업기간, 입찰마감, 제안서 제출기한, 무상유지보수기간, 하자담보책임기간을 분리해야 합니다.

사업기간 / 일정계약일로부터 N개월, 착수일, 완료일, 입찰마감, 제출기한

제출서류입찰서, 제안서, 가격제안서, 발표자료, 증빙서류 중복 정리
평가기준기술/가격 배점, 정량·정성 평가, 협상 기준, 동점자 처리
과업범위구축, 고도화, 유지관리, 데이터/AI, 보안, 시스템 연계 요구사항

이 특징 때문에 RFP corpus는 단순 chunk text만으로는 부족합니다. 원문 근거는 text/table에 남기고, 반복 질문에 필요한 후보값은 fact_candidates로 한 번 더 모아야 합니다.

3. JSON 전체 구조

Chroma에는 큰 원문 전체를 넣지 않고, 검색에 필요한 content와 작은 metadata만 넣습니다.

{ "chunk_id": "doc_c676..._fact_0001", "evidence_id": "EV_8118e4fb98", "doc_id": "doc_c6761a6fb2b7", "doc_key": "고려대학교_차세대...", "canonical_doc_id": "...", "source_file": "고려대학교_...pdf", "source_format": "hwpx | pdf", "chunk_type": "text | table | fact_candidates | toc", "embed_enabled": true, "content": "검색에 넣을 짧은 본문", "metadata": { ... 답변 정책과 문서 정보 ... }, "source_ref": { ... 원문 lookup key ... } }

역할 분리

영역역할
content임베딩되는 검색 본문입니다. 문서명, 섹션, 유형 헤더가 붙어 retrieval이 문맥을 잃지 않게 합니다.
metadata검색 후 필터링, 질문 유형별 context 조립, 답변 사용 가능 여부 판단에 쓰는 작은 정보입니다.
source_ref긴 원문, 표 원형, 상세 근거를 나중에 조회하기 위한 연결 key입니다. 기본 공유 단계에서는 source store를 열지 않습니다.
evidence_idLLM 답변의 citation과 연결할 수 있는 근거 단위 ID입니다.

왜 이렇게 길게 보존하나?

LLM이 근거를 잃지 않게 하기

문서명, 사업명, 발주기관, 섹션, fact_type을 content와 metadata 양쪽에 남겨 검색 결과가 잘려도 최소한의 문맥을 유지합니다.

왜 정책 필드가 필요한가?

숫자 오답 방지

사업예산, 실적 기준 금액, 지급조건 금액을 모두 검색할 수 있게 하되, 최종 답변에는 허용된 fact만 쓰도록 제한합니다.

팀원이 실제 코드에서 가장 많이 보게 될 필드 위주로 정리했습니다.

5. chunk 유형별 예시

같은 문서라도 검색 목적에 따라 text, table, fact 후보로 나뉩니다.

text

본문 흐름

사업 배경, 요구사항, 추진 목표처럼 문장 맥락이 필요한 질문에 사용합니다.

chunk_type: text section_path: 사업 개요 content: 사업기간, 사업예산, 추진범위...
table

표 구조

행과 열의 관계가 중요한 RFP 표를 문자열로 뭉개지 않고 table chunk로 분리합니다.

chunk_type: table table_role: retrieval_signal section_path: 제안요청 내용
fact

핵심 후보값

예산, 기간, 제출서류처럼 바로 답변에 쓰일 가능성이 높은 값을 별도 후보로 둡니다.

chunk_type: fact_candidates fact_type: project_budget answer_policy: allow_as_project_budget
identity

문서 표지판

문서 회수를 돕지만 숫자나 날짜의 최종 답변 근거로는 쓰지 않습니다.

fact_type: document_identity retrieval_role: document_identity_anchor answer_policy: route_only_not_final_answer

6. 답변 사용 정책

검색에는 포함하되, 답변에는 제한하는 것이 이 corpus의 핵심 방어 장치입니다.

문서 식별용

document_identity는 기관명, 사업명, alias로 정답 문서를 찾는 데 씁니다. 예산, 날짜, 기간의 최종 근거로 쓰면 안 됩니다.

질문 유형 의존

threshold_budget은 자격요건 질문에는 유효하지만 사업예산 질문에는 위험합니다. 그래서 eligibility_answer_enabled만 켭니다.

최종 답변 허용

project_budget처럼 본문에서 사업예산으로 확인된 값은 budget_answer_enabled를 켜서 예산 질문 근거로 사용합니다.

Allowed

예산 질문에 사용할 수 있는 값

fact_type: project_budget answer_policy: allow_as_project_budget budget_answer_enabled: true answer_risk_level: low
Blocked

예산 질문에서 막아야 하는 값

fact_type: threshold_budget answer_policy: allow_for_eligibility_exclude_for_project_budget budget_answer_enabled: false eligibility_answer_enabled: true
예산 질문

“이 사업 예산은 얼마야?”

project_budget와 budget_answer_enabled=true인 근거만 최종 금액으로 사용합니다.

자격 질문

“10억 이상 실적이 필요해?”

threshold_budget는 사업예산이 아니라 입찰 자격요건 근거로만 사용합니다.

지급 질문

“대금은 어떻게 지급돼?”

payment_terms는 지급조건 답변에는 쓰지만, 사업 전체 예산값으로 승격하지 않습니다.

문서 찾기

“보험개발원 실손보험 사업”

document_identity는 정답 문서 찾기용 표지판입니다. 숫자 답변 근거로는 사용하지 않습니다.

7. Generation 단계 활용과 예시 프롬프트 컨텍스트

LLM에게 긴 chunk를 그대로 던지는 대신, 질문 유형에 맞는 핵심 추출값과 허용 정책을 앞쪽에 배치합니다.

질문 유형별 context 조립

예산 질문project_budget와 budget_answer_enabled=true를 우선 배치
기간 질문project_duration, maintenance_period, warranty_period를 구분
제출서류 질문submission_documents와 submission_logistics를 함께 확인
자격요건 질문eligibility, threshold_budget를 허용하되 예산 답변과 분리
multi-doc 질문doc_id 기준으로 문서를 묶어 문서별 대표 근거를 배치

LLM prompt에 들어가는 규칙

근거 제한제공된 context 밖의 지식으로 답하지 않게 합니다.
정책 준수route_only_not_final_answer는 문서 식별용이며 최종 숫자 근거로 금지합니다.
Citationevidence_id, source_file, chunk_id로 답변 근거를 연결합니다.
불확실성정책상 허용된 근거가 없으면 값을 만들지 않고 missing 정보로 남깁니다.
Backfill뒤에서 보강된 문서는 보조 근거로 낮은 우선순위를 부여합니다.
