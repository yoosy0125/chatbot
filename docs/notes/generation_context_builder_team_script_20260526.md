# Generation Context Builder 팀원 설명 대본

## 1. 시작 멘트

오늘 공유드릴 내용은 제너레이션(generation) 단계에서 검색 결과를 LLM에게 어떻게 넘길지에 대한 내용입니다.

중요한 전제부터 말씀드리면, 이 코드를 그대로 정답처럼 쓰자는 뜻은 아닙니다. 지금은 우리가 설계한 JSON key를 실제 generation에서 어떻게 활용할 수 있는지 보여주는 기준 버전입니다.

## 2. before / after 정정

제가 말하는 before는 단순히 예전 generation 결과 파일 하나가 아니라, 이전에 공유했던 `rag_recallguard_one_page.html` 설계 단계입니다.

그 문서에서는 retrieval은 넓게 가져오고, generation은 metadata policy로 방어하자는 방향을 설명했습니다.

이번 문서는 그 설계가 실제 `.py`와 `.ipynb` 코드에서 어떻게 구현됐는지 보여주는 after 문서입니다.

## 3. raw text가 짧아진 이유

수정 후 raw text가 더 짧아 보일 수 있습니다. 이건 나빠진 게 아니라 의도한 변화입니다.

예전에는 LLM이 답변뿐 아니라 citation, evidence text 같은 긴 근거까지 JSON 안에 직접 쓰게 했습니다. 그래서 raw text 안에는 답이 있어도 JSON이 중간에 잘려 최종 answer가 빈 값으로 저장되는 문제가 있었습니다.

지금은 LLM에게 짧은 답변 JSON만 만들게 하고, 근거 ID, citation, 계산값, 실패 태그는 코드가 붙입니다.

그래서 비교 포인트는 raw text가 더 길고 자연스럽냐가 아니라, 최종 answer가 안정적으로 저장되고 검수 가능한 구조가 됐느냐입니다.

## 4. 전체 흐름 설명

흐름은 다섯 단계입니다.

1. retrieval 결과와 chunk JSONL을 읽습니다.
2. 질문을 먼저 분석합니다. 예산 질문인지, 기간 질문인지, 여러 문서를 비교하는 질문인지, 하나의 질문 안에 여러 요구가 섞였는지 봅니다.
3. 질문 유형에 맞춰 context를 조립합니다. 예산 질문이면 예산 후보와 예산 근거를 앞에 두고, 제출서류 질문이면 제출서류 후보와 관련 섹션을 앞에 둡니다.
4. LLM은 조립된 context를 보고 짧은 JSON 답변을 생성합니다.
5. 후처리 코드가 근거, 계산값, 실패 태그를 붙입니다.

## 5. 주요 key 설명

`fact_candidates`는 예산, 기간, 제출서류, 자격요건 같은 핵심 후보값입니다.

후보값이라고 부르는 이유는 원문에서 자동 추출한 값이라서, 바로 정답이라고 단정하지 않고 confidence와 근거를 보고 final로 올릴지 판단해야 하기 때문입니다.

`answer_policy`와 `*_answer_enabled`는 이 값을 최종 답변 근거로 써도 되는지 표시하는 장치입니다.

예를 들어 document identity는 문서를 찾는 데는 좋지만, 예산 답변의 최종 근거로 쓰면 위험합니다.

`target_slots`는 질문에 등장한 기관명이나 사업명을 문서별로 잡아두는 구조입니다.

이게 없으면 같은 기관의 다른 사업 예산을 잘못 가져올 수 있습니다.

`intent_plan`은 한 질문 안에 여러 요구가 섞였을 때 그 요구를 나누는 장치입니다.

예를 들어 “예산은 얼마이고, 사업 성격을 요약해줘” 같은 질문은 예산 조회와 요약이 동시에 들어 있습니다.

## 6. context builder 설명

context builder는 검색 결과를 그대로 LLM에게 던지는 게 아닙니다.

검색 결과 중에서 질문 유형에 맞는 JSON key를 먼저 보고, 핵심 추출값 요약을 앞에 둔 뒤, 짧은 근거 block을 붙입니다.

이렇게 하는 이유는 LLM이 긴 context 중간 정보를 놓치는 문제를 줄이기 위해서입니다.

## 7. 개선 사례 설명

HTML에 있는 Q128 같은 사례는 raw text 자체가 틀렸던 사례가 아닙니다.

오히려 raw text 안에는 답이 있었는데, JSON 파싱이나 저장 구조 때문에 최종 answer가 빈 값이 되던 사례입니다.

그래서 이 사례의 개선 포인트는 “LLM이 더 똑똑하게 답했다”가 아니라, “답이 최종 산출물에 안정적으로 남도록 파싱과 후처리를 고쳤다”입니다.

Q386, Q486 같은 계산형 질문은 LLM에게 산술을 맡기지 않고 코드 계산값을 붙이는 방향으로 보완했습니다.

## 8. 아직 부족한 점

아직 완성된 단계는 아닙니다.

가장 큰 약점은 복합 질문입니다. 한 질문 안에 예산, 요약, 비교가 같이 들어가면 일부만 답하는 경우가 있습니다.

또 금액 타입도 더 세분화해야 합니다. 사업예산, 전체 배정액, 추정가격, 투찰한도는 모두 금액이지만 의미가 다릅니다.

그리고 target 문서 선택도 더 다듬어야 합니다. 정답 문서가 context에 있어도 비슷한 다른 문서의 값을 고르는 경우가 아직 있습니다.

## 9. 마무리 멘트

정리하면, 이번 작업은 generation 답변을 완성했다기보다, 우리 JSON 구조를 실제 generation에서 사용할 수 있는 형태로 연결한 것입니다.

이제부터는 실패 사례를 보면서 intent 분해, target 선택, 금액 타입 분리, 후처리 규칙을 하나씩 보강하면 됩니다.

팀원분들이 generation을 보실 때는 단순히 프롬프트만 보지 말고, 어떤 JSON key가 context 앞쪽에 들어갔는지, 그리고 최종 answer가 어떤 근거와 연결되는지를 같이 봐주시면 좋겠습니다.
